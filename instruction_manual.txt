TRANKR — SELF-HOSTING GUIDE
=============================
This is my personal instance of Trankr and registration is restricted to
my own account. If you'd like to use Trankr yourself, the full source is here
for you to run your own copy — for free, in a few minutes. No coding
knowledge required, just follow the steps below.


WHAT YOU'LL END UP WITH
----------------------------
Your own private Trankr, running on your own server and database, with
your own login — installable on your phone and laptop like an app.


REQUIREMENTS
----------------
- A free GitHub account (to hold your copy of the code)
- A free Render account (render.com) — runs the app
- A free Neon account (neon.tech) — stores your data
- A free Google Cloud account (console.cloud.google.com) — lets you sign in
  with Google, no email service needed at all
No credit card is required for any of these at the free tier used here.

NOTE: Trankr used to send a 6-digit code by email to verify new accounts,
but Render's free tier blocks outbound SMTP ports (25/465/587), and even
routing that email through an HTTP API (Brevo) runs into deliverability
issues depending on your sending domain. Signing in with Google sidesteps
all of it — Google verifies the email for you, no email service required.
The old email/OTP path still exists as a fallback (see the Appendix at
the end of this file) if you'd rather not use Google Sign-In.


STEP 1 — Get your own copy of the code
-------------------------------------------
On GitHub, click "Fork" on this repository (top-right of the repo page).
This creates your own copy under your account, e.g.
    github.com/YOUR-USERNAME/trankr


STEP 2 — Create a free database
------------------------------------
1. Go to neon.tech and sign up (no credit card needed)
2. Create a new project
3. Copy the connection string it gives you — looks like:
       postgresql://user:password@host/dbname
   Keep this handy for Step 4.


STEP 3 — Set up Google Sign-In (so you can actually log in)
---------------------------------------------------------------------
1. Go to https://console.cloud.google.com/apis/credentials (create/select
   any project when prompted — it's free, no billing needed for this)
2. Click "Create Credentials" → "OAuth client ID"
3. If prompted to configure the "OAuth consent screen" first, choose
   "External", fill in an app name and your email, and save — you can
   skip adding scopes/test users, this is just for your own use
4. Application type: "Web application"
5. Under "Authorized JavaScript origins", click "Add URI" and enter your
   Render URL once you know it (Step 4 gives you this) — e.g.
       https://your-app-name.onrender.com
   You can also add http://localhost:8000 here for local testing.
6. Click "Create" — copy the "Client ID" it shows you (keep it for
   Step 4). You don't need the "Client secret" for this.

If you don't know your Render URL yet, you can create the OAuth client
now with a placeholder origin and edit it later (Step 4) to add the real
one — Google lets you edit authorized origins after creation.


STEP 4 — Deploy the backend (Render)
------------------------------------------
1. Go to render.com and sign up, connect your GitHub account
2. New → Web Service → select your forked repo
3. Root directory: backend
4. Build command:   pip install -r requirements.txt
5. Start command:   uvicorn main:app --host 0.0.0.0 --port $PORT
6. Add these environment variables (see backend/.env.example for the
   full reference):

     DATABASE_URL   = (the Neon connection string from Step 2)
     SECRET_KEY     = (any random string — see note below)
     GOOGLE_CLIENT_ID = (the Client ID from Step 3)
     ALLOWED_EMAILS = (leave blank for an open instance — recommended
                        for your own self-hosted copy; set to your own
                        email, comma-separated for multiple, to lock
                        registration to yourself only)
     PYTHON_VERSION = 3.11.9 (only needed if the build fails with a
                        Python version/compilation error — pins Render
                        to a stable Python version)

   For SECRET_KEY, use any long random string. If you have Python installed,
   this command generates one:
       python -c "import secrets; print(secrets.token_hex(32))"
   If not, any long random password-like string works fine too.

7. Deploy. Render gives you a URL like:
       https://your-app-name.onrender.com

That's it — the backend also serves the app's interface directly, so this
one URL is your whole app. No separate frontend hosting needed.

IMPORTANT: go back to Google Cloud Console (Step 3) → your OAuth client
→ "Authorized JavaScript origins" → add this Render URL if you hadn't
already (must match exactly, including https://, no trailing slash).
Without this, clicking "Continue with Google" will fail with an
"origin mismatch" error.


STEP 5 — Sign in
------------------------------------
Open your Render URL and click "Continue with Google." By default
registration is open to anyone who finds your URL — set ALLOWED_EMAILS
(Step 4) if you want it locked to just your own email.


STEP 6 — Install it like an app on your phone
----------------------------------------------------
Open your Render URL on your phone's browser:
- Android (Chrome): menu → "Add to Home screen" / "Install app"
- iOS (Safari): Share icon → "Add to Home Screen"

It'll install as a real app icon and open full-screen.


NOTES
---------
- Render's free tier spins down after ~15 minutes of inactivity — the
  first request after a break takes ~30 seconds to wake up. Normal.
- Neon's free tier gives 0.5 GB of storage, which is far more than a
  personal instance will use even after years of daily use.
- Your data belongs to you — this is your own Postgres database, and you
  can export or delete it at any time from your Neon dashboard.


TROUBLESHOOTING
--------------------
- "This instance is private" error on registration → you're trying to
  register on someone else's deployed instance, not your own. Make sure
  you deployed your own fork and are using your own Render URL.
- "Continue with Google" button doesn't appear → GOOGLE_CLIENT_ID isn't
  set on Render, or the page hasn't picked up the env var yet (redeploy
  after adding it).
- Clicking the Google button shows an "origin mismatch" / "idpiframe"
  error → your Render URL isn't listed under "Authorized JavaScript
  origins" for the OAuth client in Google Cloud Console (Step 3/4), or
  it doesn't match exactly (check for a trailing slash or http vs https).
- Server unreachable → Render free tier may just be waking up from idle,
  wait ~30 seconds and retry.


APPENDIX — Optional email/OTP fallback (Brevo)
----------------------------------------------------
Trankr can still fall back to old-fashioned email+password with a 6-digit
OTP code, in case you'd rather not use Google Sign-In. This is unused by
default — if GOOGLE_CLIENT_ID is set, the Google button is all you need.

Render's free tier blocks outbound SMTP ports, so this path sends the
OTP via Brevo's HTTP email API instead of raw SMTP:

1. Go to https://www.brevo.com and sign up for a free account
2. Settings → SMTP & API → API Keys tab → "Generate a new API key" →
   copy the key it shows you (only shown once)
3. Senders, Domains & Dedicated IPs → Senders → "Add a sender" → enter
   the address you want Trankr to send from and confirm it via Brevo's
   verification email
4. Account name (top-right) → Settings → Security → Authorized IPs →
   if the "API keys" row shows "Activated", click "Deactivate for API
   keys" (prevents Brevo rejecting mail when Render's outbound IP
   changes)
5. On Render, add these env vars alongside (or instead of)
   GOOGLE_CLIENT_ID:
     BREVO_API_KEY  = (the API key from step 2 above)
     EMAIL_FROM     = (the verified sender address from step 3 above)
     EMAIL_FROM_NAME = Trankr

Troubleshooting this path specifically:
- No verification email arriving → double-check BREVO_API_KEY is correct
  and that EMAIL_FROM matches a sender you've verified in Brevo. Check
  Render logs for a clear "Brevo API error ..." message.
- Check your spam folder — first emails from a brand-new sender
  sometimes land there.
- 500 error with "Brevo API error 401" mentioning an "unrecognised IP
  address" → Brevo's IP blocking is still on; see step 4 above.
- If EMAIL_FROM is a @gmail.com (or other big-provider) address, expect
  emails to be silently deferred/dropped no matter which provider you
  use — those domains reject third-party senders by design (DMARC).
  Use a domain you actually control DNS for, or just use Google
  Sign-In instead and skip this whole appendix.
